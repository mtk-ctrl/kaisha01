### dangerouslySetInnerHTML のXSSリスク

#### 使用箇所

1. **Bubble コンポーネント** (L172)
   ```typescript
   dangerouslySetInnerHTML={{ __html: text }}
   ```
   - 入力元：`step.messages[msgIdx]` — unit1.ts の hardcoded 配列

2. **Choice ボタン** (L306)
   ```typescript
   dangerouslySetInnerHTML={{ __html: c.label }}
   ```
   - 入力元：`step.input.choices[i].label` — unit1.ts の hardcoded 配列

3. **Secret Title** (L232)
   ```typescript
   dangerouslySetInnerHTML={{ __html: `🔬 ${unit1.secretTitle}` }}
   ```
   - 入力元：`unit1.secretTitle` — hardcoded ただし ruby タグ を含む

4. **Secret Points** (L238)
   ```typescript
   dangerouslySetInnerHTML={{ __html: p }}
   ```
   - 入力元：`unit1.secretPoints[i]` — ruby タグを含む hardcoded 配列

#### XSSリスク評価

**✅ 現在のリスク: 最小限**

理由：
- すべてのデータが **ソースコード内の hardcoded 配列** から来ている
- 入力フィールドやAPI レスポンスなどの **動的データを使用していない**
- ruby タグ（`<ruby><rt>`）などのセマンティック HTML のみで、script タグなどは含まれていない

**⚠️ 将来のリスク**:
- Claude API 統合で `nextStep` 後に **動的データ** を取得する場合、入力サニタイズが必須
- ユーザー入力（親の名前設定など）を bubble に反映させる場合は `dangerouslySetInnerHTML` を避けるべき

#### 推奨事項

現在は安全だが、以下の方針を記録：
- hardcoded データのみ `dangerouslySetInnerHTML` 許可
- 動的データ（API, ユーザー入力）は `textContent` 使用
- セキュリティレビュー時に再検証


### TypeScript型の問題

#### 型安全性チェック

1. **Step 型の `input` フィールド**
   ```typescript
   input: { type: 'next'; nextStep: string | null }
        | { type: 'choices'; choices: Choice[]; nextStep: string | null }
   ```
   - Discriminated Union ✅ 正しい

2. **getStep() 関数** (L25)
   ```typescript
   function getStep(id: string): Step {
     return unit1.steps.find((s) => s.id === id)!
   }
   ```
   - **⚠️ Non-null assertion (!)** をしているが、存在しない stepId を渡したら runtime エラー
   - 現在の data では問題なし、但し将来的には守りが必要

3. **reducer 関数の default return**
   ```typescript
   return state // 無効なアクションに対して state を返す
   ```
   - ✅ 安全。無限ループを防ぐ

4. **phase の型による制御フロー**
   ```typescript
   if (phase.name === 'waiting') {
     const step = getStep(phase.stepId)
     if (step.input.type !== 'choices') return state
   ```
   - ✅ 型ガード正しい

#### 型の問題：ランク付け

| 問題 | 箇所 | 重大度 | 対応 |
|------|-----|--------|------|
| Non-null assertion in getStep | L25 | 低 | 定義されていないstepIdへの防御を加える（util関数化） |
| StepVisual の visual 型 | L134 | 低 | `visual` の値を Emotion 同様に enum 化推奨 |

#### 型改善提案（実装は次フェーズ）

```typescript
// Visualization types を const で管理
const VISUALIZATION_TYPES = ['hook-emoji', 'density-tank', 'ice-emoji', 'celebration'] as const
type Visualization = typeof VISUALIZATION_TYPES[number]

// step.visual の型を string から Visualization へ
```

**現在は運用上問題なし**。

#### 1. **TypingDots アニメーション** (L158-165)

#### 2. **Image priority** (L372)
```typescript

  src={EMOTION_SRC[emotion]}
  alt={`TANQuu ${emotion}`}
  width={hasVisual ? 120 : 180}
  height={hasVisual ? 120 : 180}
```

- **最適化**: FIRST_STEP の emotion のみ priority=true にすべき

**リスク**: 低 （パフォーマンス最適化の問題）

#### 3. **Collection 画面の処理フロー** (L224)
- `phase.name === 'collection'` で early return
- **以降の displayMsg 計算が実行されない** （設計上はOK）

#### 4. **Emotion キー の Image key** (L366)
```typescript

```
- emotion が変わるたび画像が再生成される


**リスク**: 低（現実的には問題なし、しかしベストプラクティスに反する）
