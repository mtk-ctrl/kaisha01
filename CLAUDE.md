## ④ 絶対ルール（6つ）

**A. ログはコミットメッセージ＋NOW.mdテーブル更新のみ**
- コミットメッセージに内容を書く（これがログ）
- NOW.md の「直近の完了タスク」テーブルに1行追加する
- `logs/sessions/` は廃止。`logs/decisions/` は重要な経営判断のみ
- セッション終了を待たず、コミット時点で記録完了

**B. 完了報告の前に自分で確認する**
- 「しました」は動作確認してから言う
- 確認できない場合は「①確認済みの事実 ②なぜ確認できないか」を明示
- **デプロイを伴う作業**は、`claude/*` push 後に本番URLで反映を確認する（例: 主要HTMLに期待する要素がある、CSS/JSが200）。未反映なら「デプロイ待ち／未反映」と報告する

**C. インフラ・認証情報は変更したらその場でNOW.mdを更新する**

**D. 執行部が全部やる**
- 「オーナーにやってもらう必要があります」は物理的に不可能な場合のみ

**E. 実装は規模で2段階に分けて判断する**

| 規模 | 該当する作業 | フロー |
|------|-------------|--------|
| **軽微** | バグ修正・文言変更・スタイル調整・静的アセット修正・設定変更 | Jobs判断で即実装可。コミット＋NOW.md更新 |
| **中規模以上** | 新機能・UX変更・データ設計変更・フロー変更 | フルフロー必須（下記） |

中規模以上のフルフロー:
1. Jobs が案件受け取り → メンバーへ役割割り振り
2. テツ がアイデア草案を出す
3. 割り振られたメンバー がテツと議論
4. テツ が議論まとめを Jobs＋アキ に上申
5. Jobs＋アキ がレビュー・差し戻し
6. Jobs が最終GO → テツが仕様を yuu・rou へ → 実装

- **「オーナーがお願いします」は中規模以上のGOではない。テツ草案とアキレビューを経た後のGOのみ有効。**
- **社内で結論が出ていない提案をオーナー判断に丸投げしない**（草案・議論・アキレビューをセットで出す）

**F. 実装後レビューはコードレビューと画面レビューの両方必須**
- **コードレビュー**: バグ・設計・型安全を確認
- **画面レビュー**: ユーザー体験を確認してからオーナーに報告。「コードは書いた」だけでは完了にしない

コードレビューの手順（push 前に必ず実行）:
```bash
cd tanq-app && npm run build   # ビルドエラーがないこと
```
ビルドが通らない限り push しない。

> ⚠️ **push は1回ずつ、Actions の完了を待ってから次を push する**  
> 短時間に連続して push すると複数の Actions が同時に `git push origin main` を試み、  
> `remote: fatal error in commit_refs` で後続が全滅する（Race Condition）。  
> concurrency 設定で待機するようにしたが、**根本は「1 push ずつ、完了確認してから次」を守ること**。

**画面レビューの手順は `product/screen-review.md` に必ず従う**（オーナーが「画面レビューして」と言ったら、まずこのファイルを開く）。要点だけ:

1. **SVGモックは画面レビューではない。** 実際にアプリが描画した画面を撮って Read で目視する。
2. **本番URL（`tanq-app.vercel.app`）は HTTP 403 で中身が読めない** → 使わない。
3. **標準はローカル起動 + Playwright**:
   ```bash
   # 1) playwright を入れる（ブラウザ本体は /opt/pw-browsers に同梱・DLスキップ）
   PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1 npm install playwright --no-save
   # 2) 本番相当ビルドを起動（Next.js 本体は npm run start が必要）
   cd tanq-app && npm run build && (npm run start -- -p 3000 >/tmp/next.log 2>&1 &)
   # 3) スクショ（juku はフロー込みの雛形あり）
   node scripts/screenshot-juku.mjs   # → scripts/screenshots/*.png（.gitignore 済み）
   ```
4. 撮った PNG を **Read で目視**し、違和感があれば直して撮り直す。必要なら `SendUserFile` でオーナーに見せる。
