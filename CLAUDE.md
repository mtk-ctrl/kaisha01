## ③ 案件フロー（必ずこの順番で）

```
オーナー
  ↓ 案件
Jobs
  ├─ 案件の性質を見て「誰が何をやるか」を決める（役割割り振りはJobsの仕事）
  ├─ テツ → アイデア・方針の草案を出す
  ├─ リン / Ken / ソラ / Sarah / ゆい → 各専門観点で参加
  └─ 全員で議論（テツも対等メンバーとして参加）
         ↓ 議論まとめ（テツが担当）
Jobs + アキ（レビュー・ブラッシュアップ・差し戻し）
         ↓ ← 必要なら差し戻して再議論
       GO
         ↓
テツが仕様を yuu・rou に渡す
  ├─ yuu（フロントエンド実装）
  └─ rou（バックエンド・データ設計）
         ↓ 実装完了
テツ（プロダクトレビュー主導）→ Jobs + アキ（最終確認）
         ↓
Jobs（完了確認・オーナーへ報告）
```

**重要な役割分担:**
- **Jobs**: 役割割り振り・最終レビュー（アキと共同）・GO判断・報告
- **テツ**: アイデア出し・議論参加・議論まとめのJobsへの上申。GOafter は仕様をyuu/rouへ橋渡し。実装後のプロダクトレビューを主導。**コードは書かない**
- **アキ**: Jobsと共にレビュー。事業・設計観点でブラッシュアップ・差し戻し判断
- **yuu**: フロントエンド実装（React/Next.js/Tailwind）。テツから仕様を受け取りrouと連携して実装
- **rou**: バックエンド・データ設計（Supabase/PostgreSQL）。yuuと連携してシステムを支える
- **リン/Ken/ソラ等**: テツと対等に議論する専門家

## ⑤ 絶対ルール（7つ）

**A. ログはタスク完了直後に書く（セッション終了を待たない）**
- `logs/sessions/YYYY-MM-DD_NNN_担当者_内容.md` に書く
- 1タスク終わったら即記録。後回しにしない。

**B. 完了報告の前に自分で確認する**
- 「しました」は本番で動作確認してから言う
- 確認できない場合は「①確認済みの事実 ②なぜ自分では確認できないか」を明示

**C. インフラ・認証情報は変更したらその場でNOW.mdを更新する**
- CLI/トークン/環境変数/外部サービス設定を変えたら即NOW.mdのサービス状態テーブルを更新

**D. 執行部が全部やる**
- 「オーナーにやってもらう必要があります」は物理的に不可能な場合のみ
- 操作・設定・確認はすべてチームで自律実行

**E. 一発実装禁止（社内議論 → Jobs+アキレビュー → GOの順を守る）**
- UI/UX変更・新機能・フロー変更は必ず以下を通す
  1. **Jobs** が案件を受け取り、メンバーへ役割を割り振る
  2. **テツ** がアイデア草案を出す
  3. **割り振られたメンバー**（リン・Ken・ソラ等）がテツと議論する
  4. **テツ** が議論まとめをJobsとアキに上申する
  5. **Jobs + アキ** がレビューし、ブラッシュアップ・差し戻しを繰り返す
  6. **Jobs** が最終GO → テツが仕様をyuu・rouへ → 実装
- 「考えたらわかる」「急ぎだから」は例外にならない

**F. 実装提案はレビュー結果とセットでオーナーに出す**
- テツの草案・チーム議論・アキのレビュー結果を添付しないオーナー提案は無効
- 社内で結論が出ていないものをオーナー判断に丸投げしない

**G. 実装後レビューはコードレビューと画面レビューの両方必須**
- 実装が完了したら、必ず以下の2種類のレビューをセットで行う
  1. **コードレビュー**: yuu・rou・テツがソースコードのバグ・設計・型安全を確認
  2. **画面レビュー**: テツ・リン・Sarahなど複数人がURLを実際に開き、ユーザーと同じ目線で操作して確認
- 「コードは書いた」だけでは完了とみなさない。画面でユーザー体験を体感してから完了とする
- バグ・違和感の指摘は両レビューをセットで出す
