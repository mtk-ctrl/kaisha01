# ヒント設計思想 —「考える足場」3段ばしご

> 中学受験 算数（/apps/juku）をはじめ、TANQ の全演習コンテンツで守るヒントの設計原則。
> 新単元を作るとき・既存単元を直すときは必ずこの文書に従う。
> 最終更新: 2026-06-14 | 更新者: Jobs（オーナー指摘「ヒントに急に答えが出る」を受けて全面整備）

---

## 0. 一番大事な原則（これだけは絶対）

**ヒントは答えを言わない。そして「なぜその式なのか」を説明する。**

子どもが「わかった！自分で解けた！」と感じる瞬間が最大の価値（小4が「おもしろい」と思うか＝最重要KPI）。  
ヒントの目的は「この問題を解く」ことではなく、**「同じタイプの次の問題も自力で解けるようにする」**こと。

> ❌ 最悪な例（答えバレ）
> ヒント3「1800 ÷ 120 ＝ 15分。検算：Aは60×15＝900m…合わせて1800m ◎」
> → 最終ヒントが答えと検算を丸出し。子どもは考えずに写すだけ。
>
> ❌ 悪い例（WHYのない誘導）
> ヒント2「70＋80を計算してみよう！」
> → 70と80が何で、なぜ足すのかが説明されていない。150と答えられても、次の問題では使えない。
>
> ✅ 良い例（答えなし＋WHYあり）
> ヒント1「向かい合って進むと、Aが進む分もBが進む分も間を縮める。1分で何mちぢまる？」  
> ヒント2「Aが70m・Bが80m、合わせて 70＋80＝150m ちぢまる（これが「速さの和」の意味）。」  
> ヒント3「1800mを毎分150mずつちぢめるから、かかる時間 ＝ 距離÷速さ ＝ 1800÷150。計算してみよう！」  
> → WHYが各ヒントに埋まっている。次の問題でも「向かい合い＝速さの和」「時間＝距離÷速さ」が使える。

## 1. 学習画面の流れ（前提）

```
まなぶ（introSlide：図の読み方）
  → とく（問題）
       ├ 解答を入力
       ├ まちがえる → ヒントが1枚ずつ増える（最大3枚 = ヒント1→2→3）
       ├ それでもわからない →「🙋 わからない、答えを見る」ボタン
       │     → ここで初めて【答え＋explanationText（解説・検算）】が出る
       └ 正解 → 解説（explanationText）が出る
```

ポイント：**答えと検算が出る場所は「答えを見る」ボタンと正解後の2か所だけ。**
ヒント1〜3はその手前の足場であり、ここに答え・検算を書いてはいけない。

## 2. 3段ばしごの役割分担

各ヒントは「1段＝1つの思考の動き」だけを担当する。情報を1枚に詰め込まない。

| 段 | 役割 | やること | やってはいけないこと |
|----|------|----------|----------------------|
| **ヒント1（着眼・WHY①）** | この問題はどういう構造か | 「向かい合うと両者が縮める → 速さの和」「仮定して替えると差が生まれる → 仮定法」のように、**なぜその考え方を使うか**を一言で。問い返し形式にすると尚よい。 | 数式を出す・答えに関わる数値を出す |
| **ヒント2（立式・WHY②）** | どんな式になるか＋なぜその式か | 立てる式の形を見せ、**なぜその数をその演算で結ぶか**を説明する。途中の値（速さの和・1日量など、問われていない量）は出してよい。 | 問われている量（最終答え）を出す |
| **ヒント3（最後の一歩）** | あと一手で解ける状態にする | 「◯◯mを毎分□□mずつちぢめるから、かかる時間 ＝ 距離÷速さ ＝ ◯◯÷□□。計算してみよう！」のように、**なぜその計算をするか**を短く添えて終える。 | 答えの数値・単位・検算を書く |

### 各ヒントに必ず入れる「WHY」の種類

| 算数単元 | ヒント1のWHY | ヒント2のWHY | ヒント3のWHY |
|---------|------------|------------|------------|
| 旅人算（出会い） | 向かい合うとお互いが縮める | AもBも縮める → だから足す（速さの和） | 距離÷速さ＝時間 |
| 旅人算（追いつき） | 同じ向きは「速い方だけ縮める分」 | 速い方−遅い方 = 速さの差 | 差÷速さの差＝時間 |
| 流水算 | 下り=流れが背中を押す（＋）、上り=流れに逆らう（−） | 下り＋上り = 2×静水時 → ÷2で静水時 | 計算の意味 |
| 仕事算 | 毎日同じペースで仕事 → 全体÷日数＝1日量 | 2人同時だから1日量はたし算 | 全体÷1日量＝日数 |
| 鶴亀算 | 全部一方と仮定する（差が生まれる） | 1個替えると差が「1匹あたりの差」ぶん変わる | 全体の差÷1匹の差＝匹数 |
| 和差算 | 小さい方を基準に線分図 | 合計から差を引く＝2×小さい方になる理由 | 2で割れば小さい方、＋差で大きい方 |
| 相当算 | 全体を①（最小公倍数の区画数）に分ける | 区画数÷実数数＝1区画の大きさ | 全体区画数×1区画＝全体 |
| 差集め算 | 1個あたりの差が個数ぶん集まって全体の差に | あまり・不足の組み合わせパターン | 全体の差÷1個あたりの差＝個数 |

### ヒント3の書き方テンプレ
- `あとは 〔式〕 を計算してみよう！`
- `〔途中の値〕 がわかったね。〔最後の操作〕 すれば答えだよ。やってみよう！`
- 末尾は「計算してみよう」「やってみよう」「すると？」「＝ ?」のいずれかで止める。
- **数値で `＝ 15分` のように言い切らない。**

## 3. 検算（確かめ算）の置き場所

検算は中学受験で超重要。だが**ヒントには書かない**（答えがバレるため）。
検算は `explanationText`（答えを見る・正解後にだけ出る）に必ず入れる。

```
explanationText: '出会い算は「速さの和」。1800 ÷（60＋60）＝ 15分。
  たしかめ：Aは60×15＝900m、Bも900m、合わせて1800m ◎'
```

旧データで検算がヒント3にしかなかった場合は、explanationText へ移す（消さない）。

---

## 4. 図（diagram）も同じ哲学

- 問われている量は、解答前は図に **「?」** で表示する（答えバレ防止）。
- 実数を図に出すのは `showValues: true`（reveal／正解後／introSlide）のときだけ。
- ヒント回数（wrongCount）に連動して段階的に図を開示する単元（損益算・濃度算の面積図など）も、
  **問われている量だけは最後まで「?」**、解放するのは答えを見た後。

## 5. 模範単元（次に生かす「いいところ」）

全12単元・151問を棚卸しした結果、**濃度算（order8）だけ**が最後まで答えを言わない設計を貫けていた。これを唯一の手本＝標準とする。

- **濃度算（order8）★模範**：全13問、`30 ÷ 200 ＝ 0.15。これを % にすると？` のように最後の一歩を残す。
  多段階問題でも `食塩 36g。36 ÷ 0.09 ＝ 400g。400 − 300 ＝ ?` と末尾を `?` で止める手本。
  途中の値（食塩の重さなど、問われていない量）は出すが、**問われている量だけは最後まで言わない**さじ加減が見事。

### レビューで分かった「直すべきだったところ」（133問 / 151問）
- 旧設計の最終ヒントは `1800 ÷ 120 ＝ 15分。検算：…◎` のように **答え＋検算を丸出し**にしていた（旅人算・流水算・仕事算など）。
- つるかめ算は `亀2匹、鶴3羽。確かめ：…` と **答えを言葉で言い切って**いた（「を計算しよう」型に見えて実は模範ではなかった）。
- 単位変換・植木算など初期単元も `3 × 1000 ＝ 3000m！` と答え直書き。

→ すべて濃度算の書き味（最後の一歩を子どもに残す）に統一する。この一括統一を 2026-06-14 に実施。

## 6. 新単元を作るときのチェックリスト

- [ ] ヒント1は計算せず「着眼／問い返し」になっているか
- [ ] ヒント2は立式・方針までで、問われている量を言っていないか
- [ ] ヒント3は「計算してみよう／＝?」で止まり、答えの数値を言っていないか
- [ ] 検算は explanationText に入っているか（ヒントに検算を書いていないか）
- [ ] 図の「問われている量」は解答前 `?`、reveal でだけ実数か
- [ ] Ken が全問の数値・検算を検証したか
- [ ] 全文ふりがな（RubyText）が付いているか

> このチェックリストを通らないコンテンツは公開しない。
