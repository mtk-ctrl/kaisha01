# 作問メソッド — 全教科・全単元の問題とヒントの作り方

> TANQ の演習コンテンツ（歴史・地理・公民・理科・算数・国語…）の**問題とヒントを作る／既存を直す**ときの、唯一の方法論。
> ヒント文の細かい書き方（3段ばしご・検算の置き場所など）は `hint-design.md` を参照。本書はその**上位**にあたる。
> 最終更新: 2026-06-16 | 更新者: Jobs（AI CEO）

---

## 0. 目的 → 目標 → 手段（これだけは絶対に動かさない）

判断に迷ったら必ず最上位に戻る。**「これは学習者の暗記量・理解を増やすか？」** それだけが基準。

```
【目的｜なぜ・最上位・不変】
  学習者の暗記量を増やし、理解を深める
  （＝小4が「おもしろい」・親が課金したくなる）
        ↓ これに奉仕するためだけに
【目標｜何を】
  良問を増やす／弱い問題（特にヒント）を直す／学習者の害になる問題だけ捨てる
        ↓ そのための道具にすぎない
【手段｜どうやる（道具。守ること自体は目的ではない）】
  思考パターン分類・作問カルテ・配合の目安・赤チーム・トリアージ
```

> ⚠️ **手段の目的化を禁ずる。** 「配合（割合）に合わないから良問を捨てる」は本末転倒。
> 配合は“足りない種類を見つけて足す”ための健康診断であって、良問を減らす道具では絶対にない。
> 良問は割合に関係なく**全部残す**。問題数は基本**増える**方向（学習サービスなのだから）。

## 1. 養うべき6つの思考パターン

問題は「どの思考を養うか」で性格が変わる。**暗記で答えてよい問いもある**（それが悪いのではない）。鎖・関係づけが肝なのは **P3・P4だけ**。

| # | パターン | 養う力 | 暗記の扱い | ヒントが渡すもの | 代表単元 |
|---|---|---|---|---|---|
| **P1** | 記憶・定着型 | 知識を正確に覚え・思い出す | **◎ 暗記で答えてよい。それが目的** | 思い出すフック（語呂・仲間分け・関連づけ） | 植物、都道府県、歴史用語・人物 |
| **P2** | 計算・手順型 | 手順を**適切な場面で**使う | 公式・手順の暗記OK／場面判断が肝 | 着眼→どの式（最後の一歩は渡さない） | 食塩水、割合、速さ、ばね |
| **P3** | 原理・思考型 | 原理から「なぜ」を導く・未知に応用 | 最小限 | 原理の軸 | てこ、浮力、ふりこ、回路 |
| **P4** | 因果・比較・分類型 | 「なぜ・どう違う・どの仲間」で結ぶ | **事実の暗記は前提でOK**、その上で関係づけ | 比べる軸・因果の鎖の一輪 | 歴史の因果、地理、現象説明 |
| **P5** | 資料読み取り型 | グラフ・地図・史料を読み解く | 前提知識は要るが**答えは資料の中** | どこを見るか・何と比べるか | 統計、雨温図、地形図、史料 |
| **P6** | その他・自由思考型 | 型にはまらない発想・**考える喜び** | 関係なし | 問いを**開く**（答えを狭めない） | 単元を問わず |

> **P6ルール**：どの単元にも最低1問。中でも「**直接思考してよい問題**」（正解一つに縛らず、子どもが自由に考え・あそべる問い）を必ず含む。＝作り手が型に固定されるのを防ぐ枠＆「小4がおもしろい」を狙う実験枠。

## 2. 単元ごとの作問プロセス

一気にやらない（**細切れ・数問ずつ**・質＞スピード）。1単元ずつ回す。

```
Step0  単元を選ぶ
Step1  出題の配合（割合）の“目安”を置く  ← 暗記100%等に偏らないための診断。守る規則ではない
Step2  1問ずつ、作問カルテで作る／直す
Step3  パターン別の合格テスト → 赤チーム監査 → オーナー抜き取り → 学習者データ
```

### 配合の目安の使い方（重要）
- 配合は「この単元、理解を問う問題がゼロでは？」のような**偏りを見つける物差し**。
- 偏りが見えたら **足りない種類を“追加”する**（例：理解問題を足す、P6を足す）。
- **良問を“間引く”ためには使わない。** 完全な重複（同一論点の焼き直し）だけは整理してよいが、それは「配合」ではなく「重複」を理由に捨てる。

### 配合の例（食塩水・目安）
手順P2が中心・記憶P1少々・原理/因果/資料を各少々・**P6を必ず1問**。単元が変われば配合も変わる（植物ならP1中心、てこならP3中心）。

## 3. 作問カルテ（1問ごと・書いてから作る）

埋まらない＝その問いが弱い、というセンサーになる。④⑤⑦は**選んだパターンの基準**で書く。

| 項目 | 中身 |
|---|---|
| ① ねらい | この問いで何を覚えさせ／理解させたいか（1文） |
| ② パターン | P1〜P6のどれか |
| ③ 暗記／理解の線引き | このパターンで暗記してよい所・考えさせる所を分ける |
| ④ 品質基準 | **パターン別**：P1=覚える価値・フック・正確さ／P2=場面判断・正確さ／P3P4=鎖・関係づけ／P5=資料依存／P6=思考が広がるか |
| ⑤ ヒント | パターンの「渡すもの」に従う（詳細は hint-design.md） |
| ⑥ 想定誤答・つまずき | どこで間違うか。誤答選択肢はそれを反映 |
| ⑦ 合格テスト | パターンの肝：P3P4P6=盲解き＋転移／P1=正確さ＋フック／P2=数値転移／P5=資料なしで解けないこと |

### 合格テストの中身
- **盲解き**：正解を隠しヒントだけ渡す。それで答えが出たら、ヒントが思考を奪っている（P3P4で必須）。
- **操作テスト**：答えに要る頭の動きは？「思い出す」だけ＝P1。「比べる・因果・分類・導く」＝P3P4。
- **転移テスト**：聞き方を少し変えても、同じ理解で解けるか。その文だけ暗記で解けるなら理解になっていない。

## 4. 既存資産のトリアージ（棚卸し）

既存の問題には、良問・弱いヒントの良問・悪問が混在する。1単元ずつ、4判定で回す。

| 判定 | 条件 | ねらい |
|---|---|---|
| **残す KEEP** | 学習者の役に立つ良問 | **割合に関わらず全部残す** |
| **ヒント直す FIX** | 問いは良い・ヒントが弱い／パターン不一致（最多・最安） | 活かして回収 |
| **捨てる KILL** | 学習者の害になる問題だけ（事実誤り・瑣末・混乱・**完全重複**） | きれいにする |
| **足す ADD** | 配合で見えた不足（理解問題・P6 など） | 偏りを埋める |

> 結果：悪問が消え・弱いヒントが直り・足りない種類が増える。**総数はむしろ増える**＝学習者の暗記量と理解が増える方向にしか動かさない。

棚卸しの工程：
1. **Phase1**：既存問題をP1〜P6に分類し、現状配合とギャップを見る
2. **Phase2**：1問ずつ KEEP/FIX/KILL を判定（赤チーム独立監査）
3. **Phase3**：不足を ADD（特にP6必須）
4. **Phase4**：build＋画面確認＋オーナー抜き取り → 次の単元へ

## 5. 品質パイプライン（質の生命線）

作問者と検証者が同じ知能だと盲点を共有する。独立性を重ねて担保する。

- **① 作問（Jobs）**
- **② 赤チーム監査（独立）** — 別エージェントで起動し、(a)作者の理由づけを見せず問題文だけ渡す＝アンカリング回避、(b)**web典拠（教科書・過去問・公的資料）に必ず当てる**＝証拠源を分ける、(c)「壊す役」に徹し 9-3 ルーブリックで攻撃。ルーブリック＝記憶か理解か／思考を奪っていないか／理解を問えているか／第2の正解／事実精度／答えバレ・ふりがな。
- **③ 不一致プロトコル** — 赤チームがFIXを出したら、作者は**修正するか典拠で反論**。決着しなければ**オーナーへエスカレーション**（黙って押し切らない）。
- **④ オーナー抜き取り判定**
- **⑤ 学習者データで実証** — 転移正答率・即答率（考えずに即答＝漏れ）・離脱・「おもしろい?」。**“本当に理解が深まったか”を答えられるのは最後はデータと人**。赤チームはその前の足切り。

## 6. これだけは守る（運用）

- 判断は常に §0 の最上位目的に戻す。
- 1単元・数問ずつ。build → 画面確認 → オーナー判定をワンセットで。
- 全文ふりがな（RubyText）必須。
- ヒント文の作り方の詳細は `hint-design.md`。
